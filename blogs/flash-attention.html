What is Flash Attention?

April 18th, 2026

Flash Attention is one of the most widely used techniques in modern LLM training and inference. In this post, we explore the algorithm from first principles to understand why it matters. Based on Dao et al., 2022.

Problem Setup

In a transformer, self-attention operates on N vectors each of dimension D. N is the sequence length (e.g., 1024, 8192) and D is the per-head feature dimension (e.g., 64). Before any attention computation, we first project these N vectors through three weight matrices to obtain the query, key, and value matrices:

$$\begin{aligned} Q &= X W_Q \in \mathbb{R}^{N \times D} \\[6pt] K &= X W_K \in \mathbb{R}^{N \times D} \\[6pt] V &= X W_V \in \mathbb{R}^{N \times D} \end{aligned}$$ 
import torch

N = 128   # sequence length
D = 32    # feature dimension

input_feats = torch.randn(N, D)

W_q = torch.randn(D, D)
W_k = torch.randn(D, D)
W_v = torch.randn(D, D)

Q = input_feats @ W_q
K = input_feats @ W_k
V = input_feats @ W_v

Softmax and Safe Softmax

The attention output is computed as:

$$O = \text{Softmax}(QK^\top) \cdot V$$

where the softmax is applied row-wise to the N×N score matrix S = QKᵀ. For a single row x, standard softmax is:

$$\sigma(x)_i = \frac{e^{x_i}}{\sum_j e^{x_j}}$$

The problem: if any value in x is large (say, 1000), then exp(1000) overflows to infinity in FP16/FP32. The fix is safe softmax. We subtract the row maximum m = max(x) before exponentiating:

$$\sigma(x)_i = \frac{e^{x_i - m}}{\sum_j e^{x_j - m}}, \quad m = \max(x)$$

This is numerically identical to the original. Subtracting the same constant from numerator and denominator cancels out, so the largest exponent is now exp(0) = 1 and there is no overflow.

def softmax(x, dim=-1):
    m = x.max(dim=dim, keepdim=True).values
    e = torch.exp(x - m)
    return e / e.sum(dim=dim, keepdim=True)

Standard Attention and the Memory Wall

Standard attention materializes S and P as full N×N matrices. These are computed in SRAM (fast on-chip memory), but they are too large to keep there, so they must be written out to HBM (GPU high-bandwidth memory, the large but slow off-chip DRAM), then read back again for the next step.

def standard_attention(Q, K, V):
    # load Q, K from HBM → compute → write S to HBM
    S = Q @ K.T           # [N, N]
    # load S from HBM → compute → write P to HBM
    P = softmax(S, dim=-1)  # [N, N]
    # load P, V from HBM → compute → write O to HBM
    O = P @ V             # [N, D]
    return O

That's six HBM transfers total, and four of them move O(N²) data (writing S, reading S, writing P, reading P). At large sequence lengths this is the bottleneck: the GPU cores sit idle while data crawls across the memory bus.

Core Insight

Flash Attention's main goal is to eliminate the large O(N²) HBM transfers that dominate standard attention. Instead of writing and reading back full N×N matrices for S and P, it processes small tiles that fit entirely in SRAM. Each block pair triggers a compact tile load and a compact tile write — many more transfers in total, but each one moves a tiny fraction of the data, keeping the HBM bus busy with work proportional to N rather than N².

Flash Attention: Computing in Blocks

Flash Attention tiles Q, K, V into blocks of B rows each, where B is chosen so that a few blocks fit in SRAM simultaneously. We maintain three running accumulators that are updated incrementally as we process each block pair, so we never need a full N×N matrix in memory:

  • O ∈ ℝ^(N×D), the output accumulator, initialized to zeros
  • M ∈ ℝ^N, the running row-maximum, initialized to −∞
  • L ∈ ℝ^N, the running row-sum of exponentials, initialized to zeros

The outer loop iterates over j (blocks of K and V) and the inner loop over i (blocks of Q). For each pair (i, j), the tiles Q_i, K_j, V_j are loaded into SRAM. A small B×B score block S_ij = Q_i · K_jᵀ is computed entirely in SRAM, and the three accumulators for row block i are updated before moving on.

The key question is:

How do you compute softmax without having seen all N rows?

The Online Update Equations

For each block, we compute a local maximum m_ij and a local row-sum l_ij. We then merge them with the running statistics using rescaling factors:

$$\begin{alignat}{2} &M_i^{\text{new}} &&= \max(M_i,\; m_{ij}) \\[10pt] &L_i^{\text{new}} &&= \alpha \cdot L_i + \beta \cdot l_{ij} \\[10pt] &O_i^{\text{new}} &&= \frac{1}{L_i^{\text{new}}} \left( \alpha \cdot L_i \cdot O_i + \beta \cdot P_{ij} V_j \right) \end{alignat}$$

where α and β are

$$\begin{alignat}{2} &\alpha &&= \exp(M_i - M_i^{\text{new}}) \\[10pt] &\beta &&= \exp(m_{ij} - M_i^{\text{new}}) \end{alignat}$$

α and β are rescaling factors that keep the partial sums consistent as the running maximum M_i grows. Because softmax divides by a sum of exponentials anchored to the row maximum, every time a new block raises M_i, all previously accumulated values become stale — they were computed relative to the old maximum. α corrects for this by rescaling the old running sum L_i and output O_i to be expressed relative to M_inew. β plays the symmetric role for the current block, converting its local exponentials (computed relative to m_ij) into the same reference frame. If a new block does not raise the maximum, α = exp(0) = 1 and nothing needs adjusting.

Connection to Safe Softmax

This is the same idea as safe softmax, applied online one block at a time rather than to the full row at once. The running maximum M_i plays the same role as m = max(x) in the single-vector case, and the rescaling keeps the partial sums consistent as that maximum grows.

Implementation

def flash_attention(Q, K, V, B=8):
    N, D = Q.shape
    N_blocks = N // B

    M = torch.full((N, 1), float("-inf"))
    L = torch.zeros((N, 1))
    O = torch.zeros((N, D))

    for j in range(N_blocks):
        K_j = K[j*B : (j+1)*B]
        V_j = V[j*B : (j+1)*B]

        for i in range(N_blocks):
            Q_i = Q[i*B : (i+1)*B]
            O_i = O[i*B : (i+1)*B]
            M_i = M[i*B : (i+1)*B]
            L_i = L[i*B : (i+1)*B]

            S_ij = Q_i @ K_j.T                              # [B, B]
            m_ij, _ = S_ij.max(dim=-1, keepdim=True)     # [B, 1]
            p_ij = torch.exp(S_ij - m_ij)                  # [B, B]
            l_ij = p_ij.sum(dim=-1, keepdim=True)          # [B, 1]

            M_i_new    = torch.max(M_i, m_ij)
            prev_scale = torch.exp(M_i - M_i_new)
            curr_scale = torch.exp(m_ij - M_i_new)

            L_i_new = prev_scale * L_i + curr_scale * l_ij
            O_i_new = (1 / L_i_new) * (L_i * prev_scale * O_i + curr_scale * p_ij @ V_j)

            O[i*B : (i+1)*B] = O_i_new
            L[i*B : (i+1)*B] = L_i_new
            M[i*B : (i+1)*B] = M_i_new

    return O
A Note on Speed

This implementation is correct but runs slowly. The nested Python loop triggers separate PyTorch dispatch for every block pair, so the overhead dominates any I/O savings. In a real CUDA implementation the entire algorithm is fused into a single GPU kernel, which eliminates that overhead and turns the reduced HBM traffic into actual speedup. See the flash-attn library for a production kernel.

Verifying against the standard implementation:

O_std   = standard_attention(Q, K, V)
O_flash = flash_attention(Q, K, V, B=8)
print(torch.allclose(O_std, O_flash, atol=1e-5))

The result is True.

What This Implementation Leaves Out

This tutorial covers the core memory-saving mechanism of Flash Attention. The full system adds several important components on top:

  • Causal masking. In autoregressive models, each token may only attend to past tokens. This requires masking out future positions in each S_ij block before the softmax. Flash Attention handles this by skipping blocks that lie entirely below the diagonal and partially masking those that straddle it.
  • Multi-head attention. In practice, attention is computed across multiple independent heads in parallel, each with its own Q, K, V projections. Flash Attention applies the same block algorithm to each head independently.
  • Real CUDA kernels. The production implementation fuses the entire nested loop into a single kernel, avoiding all Python overhead and exploiting GPU-specific optimizations like warp-level parallelism and shared memory reuse.
